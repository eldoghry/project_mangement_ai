# Kanban Board — Backend & Integration Plan

> **Status Legend:** `[ ]` = Not started · `[x]` = Completed · `[-]` = In progress

---

## Overview

The frontend is a complete Next.js Kanban MVP with localStorage-based persistence and demo auth.
This plan covers adding a real backend, Docker, OS-specific run scripts, frontend–backend integration,
and an AI assistant feature powered by OpenRouter.

Phases are meant to be committed to git one at a time after each is fully complete.
`.gitignore` files are created first (Phase 0) so every subsequent commit is clean.

## Phase 1 — Backend Foundation (TypeScript + Express + SQLite)

> Set up a production-capable REST API that mirrors the frontend data model.
> The backend will replace localStorage persistence with a real database and upgrade demo auth to JWT.

### Subtasks

- [ ] **1.1** Initialize backend project
  - [ ] `npm init -y` inside `backend/`
  - [ ] Install dependencies: `express`, `better-sqlite3`, `jsonwebtoken`, `bcryptjs`, `cors`, `dotenv`, `zod`
  - [ ] Install dev dependencies: `typescript`, `ts-node-dev`, `@types/*`, `eslint`
  - [ ] Configure `tsconfig.json` (strict mode, `outDir: dist`, `rootDir: src`)
  - [ ] Configure `package.json` scripts: `dev`, `build`, `start`

- [ ] **1.2** Project structure
  - [ ] Create `backend/src/` folder layout: `routes/`, `controllers/`, `middleware/`, `db/`, `models/`, `lib/`
  - [ ] Create `backend/src/index.ts` (Express entry point)
  - [ ] Create `backend/src/app.ts` (app factory, middleware registration)

- [ ] **1.3** Database setup (SQLite via `better-sqlite3`)
  - [ ] Create `backend/src/db/database.ts` (singleton connection)
  - [ ] Create `backend/src/db/migrations.ts` (run-on-start schema setup)
  - [ ] Schema: `users`, `lists`, `tasks` tables
    - `users`: `id`, `username`, `password_hash`, `created_at`
    - `lists`: `id`, `title`, `position`, `user_id`, `created_at`, `updated_at`
    - `tasks`: `id`, `title`, `description`, `position`, `list_id`, `user_id`, `created_at`, `updated_at`
  - [ ] Seed 5 default lists per new user on first login

- [ ] **1.4** Authentication (JWT)
  - [ ] `POST /api/auth/login` — validate credentials, return JWT
  - [ ] `POST /api/auth/register` — create new user (optional for MVP)
  - [ ] `POST /api/auth/logout` — client-side token drop (stateless)
  - [ ] Create `backend/src/middleware/auth.ts` (JWT verify middleware)
  - [ ] Store `JWT_SECRET` in `.env`

- [ ] **1.5** Lists API
  - [ ] `GET  /api/lists` — get all lists for authenticated user (includes tasks)
  - [ ] `PATCH /api/lists/:id` — rename a list

- [ ] **1.6** Tasks API
  - [ ] `POST   /api/tasks` — create task in a list
  - [ ] `PATCH  /api/tasks/:id` — update title/description
  - [ ] `DELETE /api/tasks/:id` — delete task
  - [ ] `PATCH  /api/tasks/:id/move` — move task to different list with new position
  - [ ] `PATCH  /api/tasks/reorder` — batch reorder tasks within a list

- [ ] **1.7** Error handling & validation
  - [ ] Global error handler middleware
  - [ ] Input validation with `zod` on all body/param schemas
  - [ ] Consistent JSON error response format: `{ error: string, code?: string }`

- [ ] **1.8** Update `backend/CLAUDE.md`
  - [ ] Document any new endpoints, schema changes, or architectural decisions made during this phase
  - [ ] Mark this subtask complete after updating the file

## Phase 3 — Frontend–Backend Integration

> Replace localStorage and demo auth with real API calls.

### Subtasks

- [ ] **3.1** API client setup in frontend
  - [ ] Create `frontend/src/lib/api.ts` — typed fetch wrapper with JWT header injection
  - [ ] Store JWT token in `localStorage` as `kanban_token`
  - [ ] Add `NEXT_PUBLIC_API_URL` to `frontend/.env.local`

- [ ] **3.2** Update auth flow
  - [ ] Update `authStore.ts` `login()` to call `POST /api/auth/login` and store JWT
  - [ ] Update `logout()` to clear JWT
  - [ ] Handle 401 responses globally (redirect to `/login`)

- [ ] **3.3** Update board state management
  - [ ] Replace localStorage board persistence in `boardStore.ts` with API calls
  - [ ] On login: fetch `GET /api/lists` and populate store
  - [ ] `addTask` → calls `POST /api/tasks`
  - [ ] `updateTask` → calls `PATCH /api/tasks/:id`
  - [ ] `deleteTask` → calls `DELETE /api/tasks/:id`
  - [ ] `renameList` → calls `PATCH /api/lists/:id`
  - [ ] `moveTaskOnDragEnd` → calls `PATCH /api/tasks/:id/move`

- [ ] **3.4** Loading & error states
  - [ ] Add `isLoading` state to store
  - [ ] Show spinner or disabled UI during API calls
  - [ ] Toast / inline error on API failure

- [ ] **3.5** End-to-end integration test
  - [ ] Login with real credentials → board loads from DB
  - [ ] Create, edit, delete task → persists after page reload
  - [ ] Drag task to different list → persists after page reload
  - [ ] Rename list → persists after page reload
  - [ ] Logout + login again → same board state

- [ ] **3.6** Update `backend/CLAUDE.md` if any API contract changed during integration

## Phase 4 — AI Assistant Integration (OpenRouter)

> Add an AI assistant that can answer questions about the board and move cards using natural language.

### Subtasks

- [ ] **4.1** OpenRouter backend client
  - [ ] Create `backend/src/lib/openrouter.ts` — typed wrapper around OpenRouter chat completion API
  - [ ] Use a free model (e.g. `mistralai/mistral-7b-instruct:free`)
  - [ ] Store `OPENROUTER_API_KEY` in `.env`

- [ ] **4.2** Smoke test endpoint
  - [ ] `POST /api/ai/chat` with body `{ message: "2+2" }`
  - [ ] Send as a plain prompt, return AI response text
  - [ ] Verify model responds correctly before adding board context

- [ ] **4.3** Board-aware Q&A
  - [ ] Extend `/api/ai/chat` to accept board snapshot (all lists + tasks) in request
  - [ ] Build a system prompt that includes the board state as structured context
  - [ ] AI can answer questions like "how many tasks are in progress?" or "what's in the backlog?"

- [ ] **4.4** AI-powered card movement
  - [ ] Extend prompt with instructions for structured tool-use response
  - [ ] Parse AI response for move commands: `{ action: "move", taskId, toListId }`
  - [ ] Backend validates and executes the move via existing task move logic
  - [ ] Return updated board state in response

- [ ] **4.5** Frontend AI chat UI
  - [ ] Add `AIChatPanel` component (collapsible sidebar or floating panel)
  - [ ] Input field for user message; display AI response
  - [ ] If AI response contains a move action, apply it to the board store

- [ ] **4.6** Full AI integration test
  - [ ] Test: "What tasks are in the Done column?" → correct answer
  - [ ] Test: "Move 'Fix bug' to In Progress" → card moves visually + persists

- [ ] **4.7** Update `backend/CLAUDE.md` with AI endpoint docs
